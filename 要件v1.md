# v1要件定義: 外部連携

## 0. 方針

本バージョンでは、v0のローカル中心UXを維持したまま、以下の外部連携を追加する。

* Google Calendar

  * 既存予定の読み込み
  * 確定後の書き込み
* Google Tasks

  * 未完了タスクの候補化
* Slack

  * 完成スケジュール画像の共有

本バージョンでも、**ローカルの DailyPlan を正本**とする。
Google Calendar / Google Tasks / Slack は外部投影先であり、内部状態の唯一ソースにはしない。

Google Calendar はイベント取得に `events.list`、作成に `events.insert` を提供し、Tasks API は task list と task の REST リソースを提供する。Calendar API は private extended properties をイベントに付与できるため、アプリ由来イベントの識別に使える。Slack は現在 `files.upload` ではなく、`files.getUploadURLExternal` と `files.completeUploadExternal` を使う方式が推奨されている。 ([Google for Developers][2])

---

## 1. 認証方式

### 1.1 Google

Google 連携は、**Desktop app 用 OAuth クライアント**を使う。Installed apps はシークレットを安全に保持できない前提で、system browser を開いて local redirect URI で認可コードを受け取るのが公式フローであり、埋め込み WebView は避けるべき扱いです。必要に応じて offline access を取り、refresh token を保存する。 ([Google for Developers][1])

### 1.2 Slack

Slack は Bot Token ベースで連携する。
画像アップロードには `files:write` が必要で、共有先チャンネルへ投稿するには `channel_id` を指定して `files.completeUploadExternal` を呼ぶ。 ([api.slack.com][3])

### 1.3 トークン保存

* Google refresh token
* Slack bot token
* Slack default channel id

は OS の安全な資格情報ストアに保存する。
v1では `keytar` 採用を前提にしてよい。

## 2. Google Calendar 読み込み要件

### 2.1 目的

Google Calendar に既に入っている予定を、DailyPlan のタイムライン上に**最初から貼られているブロック**として表示する。

### 2.2 取得対象

取得には `events.list` を用い、`calendarId='primary'`、`timeMin`、`timeMax`、`singleEvents=true`、`orderBy='startTime'` を基本とする。Calendar API は `timeMin` / `timeMax` による期間取得、`singleEvents`、`orderBy=startTime` をサポートしている。 ([Google for Developers][2])

### 2.3 読み込み範囲

* 当日 00:00 〜 翌日 00:00 のイベントを取得
* ただし UI には 07:00〜24:00 範囲のみ可視化
* 07:00以前に始まり当日中にかかるイベントは、07:00開始にクリップして表示
* 24:00以降に続くイベントは、24:00終了にクリップして表示

### 2.4 UI上の扱い

Google Calendar 由来ブロックは **external block** として表示する。

#### external block の性質

* 左ペイン候補ではない
* タイムライン上に初期配置される
* デフォルトでは編集不可
* 見た目はローカル block と区別する

  * 低彩度
  * 鍵アイコンまたは GCal バッジ
* クリックで「Google Calendar イベント由来」と分かる簡易詳細を出す

### 2.5 内部モデル

既存の `ScheduleBlock` を拡張する。

```ts
type BlockSource = 'local' | 'google_calendar'

type ScheduleBlock = {
  id: string
  taskCandidateId?: string
  title: string
  subTasks: string[]
  colorId: string
  startUnit: number
  durationUnit: number
  note?: string

  source: BlockSource
  externalRef?: {
    provider: 'google_calendar'
    calendarId: string
    eventId: string
  }
  locked?: boolean
}
```

### 2.6 読み込み時の衝突

Google Calendar ブロックは「既に存在する予約」とみなし、ローカル block との重なりを許容しない。
候補やローカル block を配置するとき、external block も衝突対象に含める。

### 2.7 同期タイミング

v1では push 通知は使わない。Calendar API は sync token による増分同期を提供しているが、まずは以下で十分。

* アプリ起動時
* 日付切り替え時
* ユーザーが「Google予定を再読み込み」した時

Calendar API は `nextSyncToken` / `syncToken` を使った増分同期をサポートするが、v1では full fetch でよい。将来最適化時に導入する。 ([Google for Developers][4])

## 3. Google Tasks 読み込み要件

### 3.1 目的

Google Tasks の未完了タスクを、DailyPlan の task candidate 候補として取り込む。

### 3.2 API利用

* `tasklists.list` で task list 一覧取得
* 各 task list に対して `tasks.list` 実行

Tasks API は tasklists と tasks の REST リソースを持つ。`tasks.list` は `showCompleted`, `showDeleted`, `showHidden` を制御できる。Tasks API スコープは `tasks.readonly` と `tasks` がある。 ([Google for Developers][5])

### 3.3 取得条件

未完了タスクのみを候補化するため、基本クエリは次とする。

* `showCompleted=false`
* `showDeleted=false`
* `showHidden=false`

`tasks.list` の既定では `showCompleted=true` なので、明示的に false を指定する。 completed なタスクを first-party client 起源も含めて出すには `showHidden=true` も関係するが、今回は未完了のみなので不要。 ([Google for Developers][6])

### 3.4 候補化ルール

Google Tasks 由来の task は DailyPlan の `candidates` に変換して取り込む。

変換ルール:

* `title` ← Google Task の title
* `note` ← notes
* `subTasks` ← 空配列
* `estimatedDurationMin` ← なし
* `colorId` ← デフォルト色（例: `google-task-default`）
* `source='google_tasks'`
* `externalRef.provider='google_tasks'`
* `externalRef.taskListId`
* `externalRef.taskId`

### 3.5 重複防止

同一日内で Google Tasks 再取り込みをした場合、同一 `taskId` の candidate が既に存在するなら重複追加しない。

### 3.6 候補の性質

Google Tasks 由来 candidate は

* 通常 candidate と同じようにドラッグ配置できる
* candidate 側では編集可能
* ただし Google Tasks 本体への書き戻しは v1では行わない

つまり **read-only import** とする。

### 3.7 スコープ

v1は読み込みだけなので `https://www.googleapis.com/auth/tasks.readonly` を使う。Tasks API は readonly と read/write を分けている。 ([Google for Developers][7])

## 4. Google Calendar 書き込み要件

### 4.1 目的

ユーザーが DailyPlan を確定したら、その日の **local block** を Google Calendar に反映する。

### 4.2 反映対象

書き込み対象は次のみ。

* `source === 'local'` の block
* Google Calendar 由来 block は再作成しない
* future には更新対象にしてよいが、v1では「確定時反映」だけでよい

### 4.3 書き込み先

* 基本は `calendarId='primary'`
* 将来は設定で calendarId を選べるようにするが、v1は primary 固定でよい

Google Calendar の `events.insert` は `calendarId='primary'` を許容し、作成には最低 `start` と `end` が必要。書き込みには `https://www.googleapis.com/auth/calendar` などの write scope が必要。 ([Google for Developers][8])

### 4.4 イベント作成ルール

local block から Google Calendar event を作る。

マッピング:

* `summary` ← block.title
* `description` ← subTasks と note を整形して埋める
* `start.dateTime` ← block.startUnit を当日 datetime に変換
* `end.dateTime` ← start + duration
* `timeZone` ← Asia/Tokyo をデフォルト
* `extendedProperties.private` に app 識別情報を入れる

Calendar API は event 作成時に `start.dateTime` / `end.dateTime` を使う timed event をサポートし、extended properties に hidden key-value を保持できる。 private extended properties は特定 calendarId/eventId のコピーに紐づく。 ([Google for Developers][8])

### 4.5 識別用 extended properties

各イベントには以下を付与する。

```json
{
  "extendedProperties": {
    "private": {
      "createdBy": "daily-visual-planner",
      "dailyPlanDate": "2026-03-16",
      "localBlockId": "block_xxx"
    }
  }
}
```

Calendar API は private extended properties の設定・検索をサポートする。 ([Google for Developers][9])

### 4.6 書き込み戦略

v1では **upsert ではなく replace** を採用する。

手順:

1. 当日の既存 app-created event を `privateExtendedProperty=createdBy=daily-visual-planner` で検索
2. 同日の app-created event を削除
3. 現在の local block 群を新規作成

理由:

* 初期実装が単純
* v0→v1 の移行コストが低い
* localBlockId と同期ズレを避けやすい

Calendar API は `privateExtendedProperty` による検索ができる。 ([Google for Developers][2])

### 4.7 反映タイミング

* `確定` ボタン押下時に確認ダイアログを出す
* 「Google Calendar に反映する」を選んだ場合に書き込む
* 書き込み成功後、UIに success toast を出す
* 一部失敗時は失敗件数を表示する

### 4.8 スコープ

Calendar 読み書きのため `https://www.googleapis.com/auth/calendar` を使う。読み込みだけなら readonly scopes も可能だが、v1は反映があるため write scope を採る。 ([Google for Developers][2])

## 5. Slack 共有要件

### 5.1 目的

完成したスケジュール画像を、Slack の特定チャンネルへ共有する。

### 5.2 アップロード方式

Slack は `files.upload` ではなく、以下の3段階を使う。

1. `files.getUploadURLExternal`
2. 返された URL に PNG バイナリを POST
3. `files.completeUploadExternal`

Slack は `files.upload` の新規利用を終了しており、移行先として `files.getUploadURLExternal` + `files.completeUploadExternal` を案内している。 `files:write` が必要で、`channel_id` を指定するとそのチャンネルに共有される。 ([api.slack.com][10])

### 5.3 投稿内容

* ファイル名: `daily-plan-YYYY-MM-DD.png`
* title: `Daily Plan YYYY-MM-DD`
* initial_comment: 設定可能。v1 default は `今日のスケジュールです`
* channel_id: 設定画面で指定した既定チャンネル

### 5.4 実行タイミング

* 手動ボタン: `Slackへ共有`
* 確定時オプション: `確定してSlack共有`

### 5.5 必要スコープ

* `files:write`

Slack ファイルドキュメントでは、アップロードには `files:write` が必要。 ([api.slack.com][3])

## 11. 実装順

これで進めるのが最も安全です。

### Phase 1

* Google OAuth 接続
* Google Calendar 読み込み
* external block 表示
* collision に external block を含める

### Phase 2

* Google Tasks 読み込み
* candidate 取り込み
* 重複防止

### Phase 3

* Google Calendar 書き込み
* app-created event の replace 同期
* extended properties 付与

### Phase 4

* Slack Bot Token 設定
* 画像アップロード
* default channel 共有

## 12. AIへの実装指示

* まず Google OAuth を Electron main に実装すること
* Installed app 向けの system browser + local redirect フローを使うこと
* renderer に OAuth ロジックを置かないこと
* Google Calendar 由来 block は locked で描画すること
* Google Tasks は readonly import のみで、書き戻しをしないこと
* Google Calendar export は v1では replace 戦略を採用すること
* app-created event には private extended properties を必ず付けること
* Slack 共有は `files.getUploadURLExternal` → upload → `files.completeUploadExternal` の順で実装すること
* 旧 `files.upload` を使わないこと
